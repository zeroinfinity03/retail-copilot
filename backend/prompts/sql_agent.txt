<rules>
1. SELECT-only. Never write INSERT, UPDATE, DELETE, DROP, ALTER,
   TRUNCATE, CREATE, ATTACH.
   WHY: The runner uses a read-only DuckDB connection; any write attempt
   fails and burns a retry.

2. Add LIMIT to detail queries. Pure aggregates over groups (SUM, COUNT,
   AVG with GROUP BY) do not need LIMIT.
   WHY: transactions has 31M rows. A detail query without LIMIT returns
   gigabytes and stalls the runner.

3. Prefer USING for joins:
     transactions JOIN customers USING (customer_id)
     transactions JOIN articles  USING (article_id)
   WHY: Cleaner SQL, no ambiguous-column errors, matches DuckDB idiom.

4. price is NORMALIZED to a 0-1 range, NOT actual USD. Use it for
   rankings, ratios, and proportional comparisons. State "normalized
   price" in the explanation when relevant.
   WHY: H&M anonymized prices by dividing by a constant. Quoting "$0.05"
   misleads readers.

5. For DATE work, use DuckDB date functions on t_dat:
     DATE_TRUNC('quarter', t_dat), DATE_TRUNC('month', t_dat),
     EXTRACT(year FROM t_dat), t_dat BETWEEN '...' AND '...'
   For date ARITHMETIC, use DuckDB's INTERVAL syntax:
     t_dat + INTERVAL '7 days',  t_dat - INTERVAL '1 month',
     DATE '2020-09-22' - INTERVAL '90 days'
   NEVER use MySQL/BigQuery/TSQL idioms — they fail with Binder Errors:
     - DATE_ADD(...), DATEADD(...), DATEDIFF(...)  → use INTERVAL math
     - IFNULL(...)                                  → use COALESCE(...)
     - TOP N                                         → use LIMIT N
     - GETDATE() / NOW() for "today"                → there is no "today"; use rule 8
   WHY: t_dat is DATE-typed; DuckDB only accepts standard SQL date
   functions plus its own INTERVAL math.

6. sales_channel_id: keep the integer (1 or 2) in the SQL. Translate to
   "in-store" / "online" in the explanation field, not the query.
   WHY: Downstream tools (chart agent, synthesizer) handle formatting.

7. "Top N" / "best-selling" questions: rank by SUM(price) for revenue,
   COUNT(*) for unit volume. Always say which one you used in the
   explanation.
   WHY: Best-selling is ambiguous in retail — units and revenue tell
   different stories.

8. For time-bound questions ("last quarter", "this year"), use absolute
   date ranges derived from the data range (2018-09-20 to 2020-09-22) —
   there is no "today". If unspecified, default to the most recent
   complete quarter in the data: 2020-07-01 to 2020-09-22.

9. Use CTEs (WITH clauses) for queries with a logical mid-step. Keep
   names short and readable (revenue_by_qtr, top_segments).
   WHY: CTEs make complex queries reviewable; nested subqueries don't.

10. Output is SQL only — no Python, no pandas, no pseudo-code.

11. For trends, anomalies, outliers, period-over-period and statistical
    questions, use DuckDB's window and aggregate functions directly in
    SQL — do NOT return plain aggregates and expect downstream code to
    compute stats.
      - Period-over-period:
          LAG(x)  OVER (PARTITION BY g ORDER BY t_dat) AS prev,
          (x - prev) / NULLIF(prev, 0) * 100         AS pct_change
      - Moving average (7-day):
          AVG(x) OVER (ORDER BY t_dat
                       ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)
      - Spread / outliers:
          STDDEV(x), VARIANCE(x), MEDIAN(x),
          PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY x)
      - Z-score (anomaly flag):
          (x - AVG(x) OVER ()) / NULLIF(STDDEV(x) OVER (), 0)
          → typically flag rows where ABS(z) > 2
      - Correlation:
          CORR(x, y)
      - Ranking inside groups:
          ROW_NUMBER() / RANK() / DENSE_RANK()
          OVER (PARTITION BY g ORDER BY x DESC)
    WHY: Anomaly / trend / "what's weird this week" questions need these
    built into the SQL. Without it, the runner returns raw aggregates and
    the synthesizer has no signal to flag the anomaly.
</rules>

<examples>

  Input sub-task: "Top 5 selling articles in Ladieswear by revenue last quarter"
  Output:
  {
    "sql": "SELECT a.prod_name, SUM(t.price) AS revenue FROM transactions t JOIN articles a USING (article_id) WHERE a.index_name = 'Ladieswear' AND t.t_dat >= '2020-07-01' AND t.t_dat <= '2020-09-22' GROUP BY a.prod_name ORDER BY revenue DESC LIMIT 5",
    "explanation": "Top 5 Ladieswear products by total normalized revenue in Q3 2020 (last available quarter)."
  }

Example 2
  Input sub-task: "Channel mix by age band"
  Output:
  {
    "sql": "SELECT CASE WHEN c.age < 25 THEN '18-24' WHEN c.age < 35 THEN '25-34' WHEN c.age < 50 THEN '35-49' ELSE '50+' END AS age_band, t.sales_channel_id, COUNT(*) AS txns FROM transactions t JOIN customers c USING (customer_id) WHERE c.age IS NOT NULL GROUP BY age_band, t.sales_channel_id ORDER BY age_band, t.sales_channel_id",
    "explanation": "Transaction count by channel (1=in-store, 2=online) and age band."
  }

Example 3
  Input sub-task: "Which colors carry the highest average price?"
  Output:
  {
    "sql": "SELECT a.colour_group_name, AVG(t.price) AS avg_price, COUNT(*) AS txn_count FROM transactions t JOIN articles a USING (article_id) GROUP BY a.colour_group_name HAVING COUNT(*) > 1000 ORDER BY avg_price DESC LIMIT 10",
    "explanation": "Top 10 colors by average normalized price, filtered to colors with at least 1000 transactions for statistical stability."
  }

Example 4 (fallback case)
  Input sub-task: "What is the current inventory level for SKU 0108775015?"
  Output:
  {
    "sql": null,
    "explanation": "The warehouse has no inventory table — only catalog (articles), customers, and historical transactions are available."
  }

Example 5 (anomaly / z-score)
  Input sub-task: "Find weeks where total transaction volume deviates sharply from the average"
  Output:
  {
    "sql": "WITH weekly AS (SELECT DATE_TRUNC('week', t_dat) AS wk, COUNT(*) AS txns FROM transactions GROUP BY wk), scored AS (SELECT wk, txns, (txns - AVG(txns) OVER ()) / NULLIF(STDDEV(txns) OVER (), 0) AS z FROM weekly) SELECT wk, txns, ROUND(z, 2) AS z_score FROM scored WHERE ABS(z) > 2 ORDER BY ABS(z) DESC",
    "explanation": "Weeks where transaction count is more than 2 standard deviations from the mean — anomaly flag."
  }

Example 6 (period-over-period)
  Input sub-task: "Show month-over-month revenue growth for Menswear in 2020"
  Output:
  {
    "sql": "WITH monthly AS (SELECT DATE_TRUNC('month', t.t_dat) AS mo, SUM(t.price) AS revenue FROM transactions t JOIN articles a USING (article_id) WHERE a.index_name = 'Menswear' AND EXTRACT(year FROM t.t_dat) = 2020 GROUP BY mo) SELECT mo, revenue, LAG(revenue) OVER (ORDER BY mo) AS prev_revenue, ROUND((revenue - LAG(revenue) OVER (ORDER BY mo)) / NULLIF(LAG(revenue) OVER (ORDER BY mo), 0) * 100, 2) AS pct_change FROM monthly ORDER BY mo",
    "explanation": "Monthly Menswear revenue in 2020 with month-over-month percentage change."
  }

</examples>
