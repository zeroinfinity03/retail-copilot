LANGGRAPH — TOPIC GUIDE
========================
Brief one-line explainers of every topic in langgraphdocs.txt.
Read this to understand what LangGraph offers; dive into langgraphdocs.txt
when you need code or full detail for a specific topic.

============================================================
1. THINKING IN LANGGRAPH (Mental model)
============================================================
The 5-step recipe to build any agent in LangGraph:
    1. Map the workflow into discrete steps (nodes)
    2. Decide what each step does (LLM call, data fetch, action, human input)
    3. Design the shared state (what info flows between nodes)
    4. Implement each node as a Python function (state in → state update out)
    5. Wire nodes together with edges and compile the graph
WHY IT MATTERS: This is the entry-point reading. If you only read one section,
read this — everything else is detail on these 5 steps.


============================================================
2. WORKFLOWS vs AGENTS
============================================================
Difference between two ways of using LangGraph:
  - Workflow: Fixed, predetermined path (e.g., always SQL → chart → synth).
              You decide the order.
  - Agent:    Dynamic, LLM-driven path (the LLM chooses tools and order).

Common workflow patterns documented:
  - Prompt chaining        — output of one LLM call feeds the next
  - Parallelization        — many LLM calls run at the same time
  - Routing                — LLM classifies, then picks a branch
  - Orchestrator-worker    — main LLM splits work, sub-LLMs do it
  - Evaluator-optimizer    — one LLM writes, another scores, loop until good
USED IN OUR PROJECT: Our supervisor is an Orchestrator-worker pattern.


============================================================
3. PERSISTENCE (Checkpoints)
============================================================
LangGraph automatically saves graph state after every super-step (checkpoint),
keyed by a thread_id.
USED FOR:
  - Resuming a graph that was paused (human-in-the-loop)
  - Restarting after a crash (durable execution)
  - Conversation memory across multiple user turns
  - Time-travel debugging (replay or fork from any past checkpoint)
Backends: In-memory (testing), SQLite, Postgres, Redis, MongoDB.


============================================================
4. DURABLE EXECUTION
============================================================
If your graph crashes mid-run or you stop it, it can resume EXACTLY where it
left off — not restart from scratch. Requires a checkpointer.
USED FOR: Long-running workflows that take minutes/hours and can't afford to
re-do completed work after a failure.
Three "durability modes":
  - exit  → only save at end (fastest, can't recover mid-run)
  - async → save in background (good default)
  - sync  → save before every next step (safest, slowest)


============================================================
5. FAULT TOLERANCE
============================================================
Per-node controls for handling failures gracefully:
  - retry_policy   → automatically retry failed nodes with backoff
  - timeout        → cap how long a single node attempt may run (async only)
  - error_handler  → run a compensation function after retries exhaust
  - set_node_defaults → apply the same retry/timeout/handler to ALL nodes
USED FOR: Flaky API calls, slow LLMs, network blips. Lets the graph self-heal
without manual intervention.


============================================================
6. EVENT STREAMING (v1.2+)
============================================================
The modern, recommended streaming API. Gives you typed projections like:
  - stream.messages    → token-by-token LLM output
  - stream.values      → final state snapshots
  - stream.output      → just the final answer
  - stream.subgraphs   → observe nested graph activity
  - stream.interrupts  → see when graph pauses for human input
USED FOR: Showing live progress in a UI (typewriter effect, status updates,
chart rendering as it streams).


============================================================
7. STREAMING (stream_mode API — older)
============================================================
The lower-level streaming API. Modes:
  - updates  → state changes per node
  - values   → full state snapshots
  - messages → LLM tokens
  - custom   → user-defined progress events
  - checkpoints, tasks, debug → lifecycle events
WHEN TO USE: When event_streaming doesn't expose what you need, or for older
LangGraph versions.


============================================================
8. INTERRUPTS (Human-in-the-loop)
============================================================
The `interrupt()` function pauses the graph and waits for a human's input.
USED FOR:
  - Approval gates ("approve sending this email?")
  - Review and edit ("here's a draft, edit it before sending")
  - Validation loops (re-ask if input is invalid)
  - Tool approval (pause before tool runs)
Resume with `Command(resume=<value>)`. Requires a checkpointer.


============================================================
9. TIME TRAVEL
============================================================
Two operations on past checkpoints:
  - Replay → re-run from a prior checkpoint (LLM calls fire again)
  - Fork   → branch from a past checkpoint with modified state to explore
            an alternative path
USED FOR: Debugging ("what if the user said X instead?"), creating
multiple parallel scenarios, testing branches.


============================================================
10. MEMORY
============================================================
Two distinct concepts both called "memory":
  - Short-term memory  → conversation history within ONE thread
                         (uses checkpointer, thread_id)
  - Long-term memory   → cross-thread data ("user prefers dark mode")
                         (uses a Store, namespaced by user/context)
Long-term memory supports semantic search (vector embeddings).
Includes helpers to trim/summarize/delete old messages to control token cost.


============================================================
11. SUBGRAPHS
============================================================
Use a whole graph as a single node inside another graph. Two ways to wire:
  - Call inside a node → parent and subgraph have DIFFERENT state schemas;
                         wrapper function transforms state in/out
  - Add as a node      → parent and subgraph SHARE state keys; no wrapper
                         needed
Subgraph persistence modes:
  - Per-invocation (default) → fresh state per call, supports interrupts
  - Per-thread             → subgraph remembers across calls (multi-turn)
  - Stateless              → no checkpointing (fastest, no resume)
USED FOR: Multi-agent systems, reusable workflow components, team boundaries.


============================================================
12. APPLICATION STRUCTURE
============================================================
The file/folder layout LangGraph expects for deployment:
  - langgraph.json        → config file (which graphs, dependencies, env)
  - graphs                → your Python files defining the graphs
  - requirements.txt /
    pyproject.toml        → dependencies
  - .env                  → API keys and env vars
USED FOR: Deploying agents to LangSmith Deployment or similar managed
hosting.


============================================================
13. FRONTEND PATTERNS (React / Vue / Svelte / Angular)
============================================================
Pre-built `useStream` hook for connecting a frontend to a LangGraph
backend. Gives you:
  - Live token streaming into the UI
  - Per-node status cards (which node is running, which finished)
  - Interrupt handling (pause UI for human input)
  - Resume after interrupt with `Command(resume=...)`
USED FOR: Building chat UIs that show real-time agent progress.


============================================================
14. GRAPH API OVERVIEW (Core concepts)
============================================================
The three building blocks of every LangGraph:
  - State    → the shared data dict (or Pydantic model) every node reads/writes
  - Nodes    → Python functions that read state and return state updates
  - Edges    → connections between nodes; can be static, conditional, or
              dynamic via Command/Send
Other key concepts here:
  - StateGraph    → the main graph class
  - Reducers      → control how state updates merge (e.g., append vs replace)
  - MessagesState → built-in state class for chat history
  - START / END   → special entry/exit nodes
  - Send API      → dynamic fan-out for map-reduce patterns
  - Command       → combine state update + routing in one return value
  - Recursion limit → max super-steps before raising GraphRecursionError
                      (default 1000)
USED IN OUR PROJECT: All of these except Send (we use list-return conditional
edges for fan-out, which is simpler).


============================================================
15. USE THE GRAPH API (Practical guide)
============================================================
Hands-on examples covering:
  - Defining and updating state
  - Reducers (`Annotated[list, operator.add]` style)
  - Overwrite reducer (bypass reducer when you need to)
  - Different input/output schemas
  - Private state between nodes
  - Pydantic state schemas (with runtime validation)
  - Runtime configuration (context_schema for per-run config)
  - Retry policies, timeouts, error handlers, node caching
  - Sequences (linear node chains)
  - Branching (parallel nodes that fan in)
  - Deferred execution (wait for all branches before running a node)
  - Conditional routing
  - Map-reduce with the Send API
  - Loops with termination conditions
  - Async (use async def + await for IO-bound nodes)
  - Command for combined state-update + routing
  - Subgraph navigation
  - Visualization (draw_mermaid_png)
USED IN OUR PROJECT: We use sequences, branching/parallel, conditional
routing, Pydantic schemas (for ChartSpec), and visualization.


============================================================
16. AGENT CHAT UI
============================================================
A drop-in Next.js chat UI that talks to any LangGraph agent over the
standard endpoint. Handles message streaming, tool-call rendering,
interrupts, and time-travel debugging out of the box.
USED FOR: Quick prototyping — you point it at your backend and instantly
have a chat interface.
WHY WE DIDN'T USE IT: We have a custom React UI for this project. Agent
Chat UI would be the right choice for a quick MVP.


============================================================
QUICK MAP — WHAT WE ACTUALLY USE IN OUR PROJECT
============================================================
✅ Used today:
   - Graph API overview (StateGraph, nodes, edges, conditional edges)
   - Use the Graph API (parallel branching, Pydantic schemas)
   - Streaming (we wrap and serve responses via FastAPI)
   - Workflows pattern (orchestrator-worker = our supervisor)

📌 Used implicitly (LangGraph handles it for us):
   - Persistence (turned off; we don't have a checkpointer in v1)
   - Fault tolerance (default retry behavior on node failures)

⏸ Not used yet, but useful for v2:
   - Interrupts          → for human approval before queries hit production DB
   - Memory              → multi-turn chat history (currently each request is
                          independent)
   - Subgraphs           → if we add specialized sub-flows (e.g., a separate
                          "forecasting agent" with its own multi-step graph)
   - Event streaming     → richer token-by-token UI updates
   - Time travel         → debug or replay specific agent runs

❌ Not relevant for this project:
   - Durable execution    → only matters for very long-running graphs
   - Graceful shutdown    → only matters for server-managed workloads
   - Application structure → only matters if deploying to LangSmith
   - Agent Chat UI        → we built our own React UI
