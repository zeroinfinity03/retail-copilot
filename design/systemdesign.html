Retail Insights Copilot — System Design

Introduction

A multi-agent assistant for retail analytics.

A user sends a query in natural language. A supervisor agent decomposes the question into sub-tasks and dispatches them in parallel to specialist agents capable of market research, code execution, generating dashboards, and doing forecasting. Once the specialists return, a synthesizer composes their outputs into one grounded, citation-backed report.

High-level design

The Agents

Six specialist agents make up the pipeline. Each has one focused responsibility, takes a typed input, and returns a structured dict. The supervisor orchestrates; the rest do the actual work.

SUPERVISOR AGENT

The orchestrator

The first agent the user's query reaches. Decides which specialists to call and in what order. It does not fetch data, run code, or write the final answer — only plans and dispatches. The structured plan it produces drives every subsequent step.

Input

JSON request body from the frontend (POST /api/chat): 
{
  "messages": [
    {"role": "user",      "content": "..."},
    {"role": "assistant", "content": "..."},
    ...
  ]
}
main.py extracts messages[-1].content (the latest user message) and passes it as user_query: str to the supervisor.

Output

state = {
  "user_query":           str,
  "plan":                 {"rationale": str, "steps": list},
  "sql_results":          dict | None,
  "web_results":          dict | None,
  "forecast_results":     dict | None,
  "chart_results":        dict | None,
  "final_report":         str | None,
  "synthesizer_skipped":  bool,
  "synthesizer_error":    str | None,
}
Each *_results slot holds that specialist's full return dict (shape shown in the agent's own Output card), or None if the specialist wasn't in the plan.

Internal flow — step by step

1
Frontend → main.py via POST /api/chat
User types a question in the chat UI. The frontend collects the full message history (current message plus every prior exchange in this session), builds a JSON body, and posts it to the backend.
// frontend/src/App.tsx
await fetch("/api/chat", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ messages }),
});
Single-turn request body — what the very first message looks like:
// First turn
{
  "messages": [
    { "role": "user", "content": "Show me top 5 best-selling Ladieswear last quarter" }
  ]
}
Multi-turn — the same array carries every prior message in the session:
{
  "messages": [
    { "role": "user",      "content": "Top 5 Ladieswear sellers last quarter" },
    { "role": "assistant", "content": "Ladieswear top 5 by revenue: ..." },
    { "role": "user",      "content": "Now project the next 90 days" }
  ]
}
If the body is malformed (missing messages, wrong types, extra fields), FastAPI auto-returns 422 Unprocessable Entity before the handler even runs — Pydantic validates the schema first.
2
main.py validates and dispatches to a worker thread
FastAPI validates the body against a Pydantic ChatRequest schema. Because the agent pipeline is synchronous, the handler ships the work onto a worker thread via asyncio.to_thread, freeing the event loop for other requests.
@app.post("/api/chat")
async def chat(req: ChatRequest) -> ChatResponse:
    user_query = [m for m in req.messages if m.role == "user"][-1].content
    state = await asyncio.to_thread(run_pipeline, user_query)
3
Supervisor initialises the shared state dict
First action inside supervisor_P.run() — allocate one slot per specialist, all starting as None. The dict is the shared bus the rest of the pipeline reads from and writes to.
def run(user_query):
    plan = make_plan(user_query)
    state = {
        "user_query": user_query,
        "plan": plan.model_dump(),
        "sql_results": None, "web_results": None,
        "forecast_results": None, "chart_results": None,
        "final_report": None,
    }
4
make_plan() — the planner LLM call
One structured-output call to gpt-5-mini. The supervisor system prompt is loaded from disk, the user query is appended, and OpenAI's structured-outputs feature forces the response into the typed Plan shape. The LLM cannot return free-form text — only a valid Plan.
response = client.chat.completions.parse(
    model="gpt-5-mini",
    messages=[{"role": "system", ...}, {"role": "user", ...}],
    response_format=Plan,
)
return response.choices[0].message.parsed   # typed Plan object
The Plan schema — what the LLM must return:
class PlanStep(BaseModel):
    agent: Literal["sql", "web", "forecast", "chart"]
    task: str

class Plan(BaseModel):
    rationale: str
    steps: list[PlanStep]
Example output — for the query "Project Ladieswear revenue for the next 90 days and tell me what the market expects for womenswear in 2026", the planner returns:
Plan(
    rationale="internal forecast and external market view are "
              "independent — fan them out in parallel",
    steps=[
        PlanStep(agent="forecast", task="Forecast Ladieswear daily revenue for the next 90 days"),
        PlanStep(agent="web",      task="Market outlook and expert projections for womenswear in 2026"),
    ],
)
5
Parallel dispatch — code / research / forecast in one superstep
The supervisor walks the plan, groups code, research, and forecast into a parallel batch (they're independent), and submits each to a ThreadPoolExecutor. As each thread finishes, its result is written into the matching state slot via as_completed. Loop exits only when all parallel futures are done.
with ThreadPoolExecutor(max_workers=len(parallel_jobs)) as pool:
    futures = {pool.submit(runners[a], task): a for a, task in parallel_jobs.items()}
    for fut in as_completed(futures):
        agent = futures[fut]
        state[f"{agent}_results"] = fut.result()
Why parallel matters — time math. If these three agents ran one after another, the total wait would be the sum of all three:
# Sequential (one after another)
t_code + t_research + t_forecast  ≈  3 + 5 + 7  =  15 seconds
With ThreadPool dispatch, total wait drops to whatever the slowest one takes — the others finish in the same window:
# Parallel (all running at once)
max(t_code, t_research, t_forecast)  ≈  max(3, 5, 7)  =  7 seconds
The slowest specialist sets the floor; nothing adds. That's the real reason these three live in the parallel batch — user-facing latency drops by 2× or more on typical multi-agent plans.
6
Visualization agent runs sequentially (only if needed)
Visualization cannot run in the parallel batch because it consumes the code agent's rows. The supervisor fires it after the parallel batch finishes, passing the code agent's rows directly. If there are no rows or visualization isn't in the plan, this step is skipped.
if needs_chart and sql_out.get("rows"):
    state["chart_results"] = run_chart(
        columns=sql_out["columns"],
        rows=sql_out["rows"],
        user_question=user_query,
    )
7
Synthesizer composes the final narrative
All specialist outputs are now in state. The synthesizer reads every slice, calls one more LLM, and returns the grounded markdown narrative. The supervisor then returns the populated state dict to main.py, which packages it into a ChatResponse for the frontend.
synth_out = run_synthesizer(state)
state["final_report"] = synth_out["final_report"]
return state   # back to main.py
CODE AGENT

SQL generation & sandboxed execution

Converts a natural-language analytics question into safe DuckDB SQL and executes it inside a sandboxed environment. The LLM never touches the database directly — Python is the only executor.

Input

task: str — a natural-language sub-task from the supervisor.
e.g., "Top 5 Ladieswear sellers by revenue, Q3 2020"

Output

{
  "sql":         str | None,
  "explanation": str,
  "columns":     list[str],
  "rows":        list[dict],
  "row_count":   int,
  "error":       str | None,
}

The warehouse it queries

The warehouse is a single DuckDB file on local disk (backend/data/db/hm.duckdb). DuckDB is the open-source, embedded, columnar OLAP engine — the local equivalent of BigQuery, Snowflake, or Redshift. It speaks standard SQL, runs millions of rows in under a second on a laptop, and ships as a single Python package — no server, no cluster, no cloud bill. Same query patterns as the cloud warehouses; just point the agent at a different connection string to swap in production.

How data is laid out — columnar storage

DuckDB stores each column separately on disk. Queries like SUM(price) GROUP BY t_dat read only the price and t_dat columns and skip the others entirely — that's the 5–15× I/O win over row-based storage when scanning a 33M-row table.

Internal flow — step by step

1
Supervisor sends a task → generate_sql() LLM call
A natural-language sub-task arrives (e.g., "Top 5 Ladieswear sellers last quarter"). The agent loads its system prompt, combines it with the task, and makes a single structured-output call that returns a typed SQLOutput object. The LLM never touches the database.
response = client.chat.completions.parse(
    model="gpt-5-mini",
    messages=[{"role": "system", ...}, {"role": "user", ...}],
    response_format=SQLOutput,
)
return response.choices[0].message.parsed
The SQLOutput schema — what the LLM must return:
class SQLOutput(BaseModel):
    sql: Optional[str]       # SELECT-only DuckDB query, or None if out of scope
    explanation: str          # one-line plain English summary (20 words max)
sql can be None — that's how the LLM declines when the task is out of scope (e.g., asking for inventory levels the warehouse doesn't have). Step 2 handles that path. explanation is always populated, even when sql=None, so the synthesizer can phrase a graceful "data not available" reply.
2
If the LLM declines (sql=None) → early return
When the question is out of scope (e.g., asking for inventory levels the warehouse doesn't have), the LLM is allowed to return sql=None with an explanation. The agent skips DuckDB entirely and returns an empty result dict so the synthesizer can phrase a graceful "data not available" reply.
if llm_output.sql is None:
    return {"sql": None, "explanation": llm_output.explanation,
            "columns": [], "rows": [], "row_count": 0, "error": None}
Why this is graceful, not an error. Notice error: None in the return — the agent is signaling "I read the question, the warehouse genuinely doesn't have this data" rather than "something broke." The synthesizer reads error: None + empty rows + a non-empty explanation and produces a user-facing message like "Inventory levels aren't tracked in this warehouse" instead of a confusing failure message.
3
Safety check + DuckDB execute
Before the SQL hits the engine, the keyword blocklist scans for write operations (DROP/UPDATE/INSERT/...) using padded matching so column names like UPDATED_AT don't false-trigger. DuckDB opens in read_only=True mode for defense in depth. On success, returned rows are capped at 1,000.
banned = contains_banned_keyword(sql)
if banned: return {"error": f"Blocked: '"{banned}"'", ...}

con = duckdb.connect(str(DB_PATH), read_only=True)
result_df = con.execute(sql).fetchdf()
4
One-shot retry on execution failure
If DuckDB raises an exception (syntax error, missing column, type mismatch), the agent calls the LLM once more with the error message as feedback. The LLM gets a chance to self-correct. Maximum one retry; if it fails again or returns sql=None, the original error is preserved.
if exec_result["error"] and MAX_RETRIES > 0:
    retry = generate_sql(task, error_context=exec_result["error"])
    if retry.sql:
        exec_result = execute_sql(retry.sql)
        llm_output = retry
The retry mechanism in detail. Two conditions gate the retry: an actual error string in exec_result["error"] AND MAX_RETRIES > 0 (default 1). When triggered, generate_sql() is re-called with error_context — the function appends the error message to the user prompt so the LLM sees both the original task and what broke last time. This is bounded: exactly one retry, never a loop.
Edge case — retry returns sql=None. If the LLM gives up on the retry (returns sql=None), the agent keeps the original error and original SQL in the result dict — it does not overwrite. The user sees the SQL that actually ran along with what went wrong, instead of a confusing "we tried but couldn't" message that hides the original failure.
5
Pack result and return to supervisor
The final dict combines the LLM's SQL + explanation with the execution result (columns, rows, row_count, error). Supervisor stores it as-is in state["sql_results"] — passed through to the chart agent and synthesizer downstream.
return {
    "sql": llm_output.sql,
    "explanation": llm_output.explanation,
    **exec_result,         # columns, rows, row_count, error
}
RESEARCH AGENT

Live market intelligence via Perplexity Sonar Pro

Competitor news, industry trends, pricing changes, consumer sentiment. Sonar Pro internally expands queries, searches the web, reads pages, and synthesizes a grounded answer with source citations.

Input

task: str — a natural-language market-research sub-task from the supervisor.
e.g., "2026 fashion retail trends"

Output

{
  "answer":             str | None,
  "citations":          list[str],
  "search_queries_run": int,
  "model":              str,
  "error":              str | None,
}

Internal flow — step by step

1
Build messages: system prompt + user task
The agent loads its system prompt from disk (which defines its role as an external market researcher) and appends the supervisor's sub-task as the user message. Standard OpenAI messages format because Sonar Pro is OpenAI-compatible.
messages = [
    {"role": "system", "content": load_system_prompt()},
    {"role": "user",   "content": task},
]
2
Call Perplexity Sonar Pro (with search controls)
The agent uses the OpenAI SDK with base_url pointed at Perplexity. The whole call is wrapped in try/except so a network failure returns an error dict instead of crashing.
Client setup — same SDK, different URL. The agent uses the OpenAI Python SDK with base_url rewritten:
_client = OpenAI(
    api_key=os.getenv("PERPLEXITY_API_KEY"),
    base_url="https://api.perplexity.ai",
)
Because Perplexity is OpenAI-compatible, every other line of code looks identical to a regular OpenAI call — same client, same chat.completions.create() shape, no special library:
response = _get_client().chat.completions.create(
    model="sonar-pro",
    messages=messages,
    extra_body={
        "web_search_options": {"search_context_size": "high"},
        "search_recency_filter": "year",
    },
)
The two extra_body parameters — Perplexity-specific options that the OpenAI SDK doesn't know about, so they ride through extra_body:
┌──────────────────────────────────────────┬────────┬───────────────────────────────────┐
│ Parameter                                │ Value  │ Effect                            │
├──────────────────────────────────────────┼────────┼───────────────────────────────────┤
│ web_search_options.search_context_size   │ "high" │ Read more pages, deeper context   │
│ search_recency_filter                    │ "year" │ Bias toward last-year content     │
└──────────────────────────────────────────┴────────┴───────────────────────────────────┘
Inside Sonar Pro (black box). Perplexity does: sub-query expansion → parallel web search → page reads → synthesis with citations. The agent doesn't see any of this — it just receives a response object with the answer + citation URLs attached.
3
Parse response — model_dump() for non-standard fields
Perplexity returns extra fields (citations, search_results, etc.) that the OpenAI SDK's typed model doesn't know about. response.model_dump() converts the Pydantic object into a plain dict so the agent can pull the answer text, citation URLs, and search-query count.
data = response.model_dump()
answer = data["choices"][0]["message"]["content"]
citations = data.get("citations") or []
num_queries = data.get("usage", {}).get("num_search_queries", 0)
4
Return result dict to supervisor
The agent returns a consistent dict shape on both the happy path and the failure path — answer, citations, search_queries_run, model, error. Supervisor stores it in state["web_results"] and the synthesizer reads it later, embedding citation URLs inline in the final narrative.
# Success path
return {
    "answer": answer,
    "citations": citations,
    "search_queries_run": num_queries,
    "model": "sonar-pro",
    "error": None,
}
Failure path returns the same keys, different values. If the Sonar Pro call raised (network error, invalid key, rate limit), the except branch returns:
# Failure path — same keys, error-shaped values
return {
    "answer": None,
    "citations": [],
    "search_queries_run": 0,
    "model": "sonar-pro",
    "error": str(e),         # the exception message
}
Why the shapes match. Downstream code (supervisor, synthesizer, frontend) never has to special-case "did web fail" by checking the dict shape — same keys are always present. Callers just check error — None means use the answer + citations, anything else means surface the error in the final report. One contract, two code paths.
FORECAST AGENT

Prophet + SARIMA ensemble forecasting

Projects future revenue or unit volume from internal warehouse history. Runs Prophet and SARIMA (AutoARIMA) side by side, scores both on a 30-day holdout, then refits both on the full series and takes an ensemble average.

Input

task: str — a natural-language forecasting sub-task from the supervisor.
e.g., "Project Ladieswear daily revenue for next 90 days"

Output

{
  "spec":              dict,
  "sql":               str,
  "series_label":      str,
  "horizon_days":      int,
  "historical":        list[dict],
  "prophet_forecast":  list[dict] | None,
  "sarima_forecast":   list[dict] | None,
  "ensemble_forecast": list[dict] | None,
  "metrics":           dict,
  "values_table":      list[dict],
  "chart_html":        str | None,
  "explanation":       str,
  "error":             str | None,
}

Internal flow — step by step

1
make_forecast_spec() — LLM parses the task into a typed ForecastSpec
A structured-output LLM call decides what historical series to fetch (sql_task), how far to project (horizon_days, capped at 180), granularity, holiday calendar (default Sweden), and a human-readable label. Free-form text is not allowed.
response = client.chat.completions.parse(
    model="gpt-5-mini",
    messages=[{"role": "system", ...}, {"role": "user", ...}],
    response_format=ForecastSpec,
)
return response.choices[0].message.parsed
The ForecastSpec schema — what the LLM must return:
class ForecastSpec(BaseModel):
    sql_task:         str                                # brief for the SQL agent
    horizon_days:     int                                # 7 - 180
    granularity:      Literal["day", "week", "month"]
    country_holidays: Optional[str] = "SE"      # Prophet holiday calendar
    series_label:     str                                # human-readable label
    explanation:      str                                # one-sentence summary
The horizon_days hard-cap at 180 lives in the schema itself (le=180) — beyond that, confidence bands explode on a 24-month series, so the agent refuses earlier rather than producing low-quality projections.
2
Fetch historical series via the SQL agent + reshape + validate
The forecast agent calls sql_agent.run(spec.sql_task) internally — supervisor doesn't pass SQL data to it. The returned rows are auto-mapped into a pandas DataFrame with ds (date) and y (numeric) columns. If the series has fewer than 90 days, the agent short-circuits with an error.
sql_out = run_sql(spec.sql_task)
df = _rows_to_dataframe(sql_out)
if len(df) < MIN_HISTORY_DAYS:   # 90 days minimum
    result["error"] = f"Series too short ..."
    return result
3
Holdout score — Prophet and SARIMA on last 30 days
Both models fit on everything except the last 30 days, then predict the held-out 30 and report MAPE. The "winner" is tagged but both models continue forward — this score is for transparency, not selection.
train = df.iloc[:-30]; holdout = df.iloc[-30:]
metrics["prophet_mape"] = _mape(holdout.y, fit_prophet(train, 30).yhat)
metrics["sarima_mape"]  = _mape(holdout.y, fit_sarima(train, 30).yhat)
Why MAPE specifically? MAPE is the Mean Absolute Percentage Error:
MAPE = mean(|actual − predicted| / |actual|) × 100
It's scale-free — the percentage is comparable across very different magnitudes. So a Ladieswear forecast (high revenue) and a Sport forecast (lower revenue) can both be evaluated with the same metric. A reading of 12% means predictions average 12% off the truth, regardless of category. Each fit is wrapped in its own try/except — if one model errors numerically, the other still produces a score.
How SARIMA is actually implemented. Inside the agent, SARIMA is wired through AutoARIMA(season_length=7) from statsforecast — same model family, but the (p,d,q)(P,D,Q,s) parameters are auto-selected by AIC instead of being hand-tuned. The weekly seasonality (season_length=7) makes sense for daily retail data: day-of-week patterns are the dominant cycle.
4
Refit BOTH models on the FULL series and project the horizon
The holdout was only for scoring. To produce the final forecast, each model is re-fit on the complete history (no holdout) and projects horizon_days into the future. Each returns a DataFrame with (ds, yhat, yhat_lower, yhat_upper).
Why refit on full data after holdout? The holdout existed only to compare the two models honestly. For the actual forecast that ships to the user, leaving out 30 days of training signal would be wasteful — especially for a 2-year series where every additional month of pattern matters. Refit on the full series, then project forward.
Independent try/except per model. Each refit is wrapped separately. If Prophet fails on the full data but SARIMA succeeds, the agent still produces a SARIMA-only forecast (ensemble is skipped). Only when both fits fail does the agent return a hard error.
5
Compute ensemble — average of the two forecasts
Per day: ensemble_yhat = (prophet_yhat + sarima_yhat) / 2. The confidence band is the union of the two bands (min of lower bounds, max of upper bounds) — intentionally wider, reflecting model disagreement as uncertainty.
ensemble_fc = pd.DataFrame({
    "ds":         prophet_fc["ds"],
    "yhat":       (prophet_fc.yhat.values + sarima_fc.yhat.values) / 2,
    "yhat_lower": np.minimum(prophet_fc.yhat_lower, sarima_fc.yhat_lower),
    "yhat_upper": np.maximum(prophet_fc.yhat_upper, sarima_fc.yhat_upper),
})
6
Build a downsampled values table for the synthesizer
Full forecast is ~90 days × 3 columns — too many tokens for the synthesizer's LLM call. The agent picks the first 10, middle 10, and last 10 rows: enough to skim trend without blowing the token budget.
7
Build the combined Plotly chart
One Plotly figure with: historical line (solid black), Prophet forecast (dashed blue) + uncertainty ribbon, SARIMA forecast (dashed orange) + ribbon, and the ensemble (bold green) on top. Exported as self-contained HTML for the frontend iframe. Returned to the supervisor along with the values table, metrics, and all three forecasts.

Future: swap to darts (optional)

The agent currently calls prophet and statsforecast directly — two libraries, two different APIs. The darts library wraps both behind a uniform TimeSeries object, so every model uses the same fit() / predict() shape.

Code today — two libraries, two shapes:

# Prophet — uses a pandas DataFrame natively
from prophet import Prophet
m = Prophet()
m.add_country_holidays(country_name="SE")
m.fit(train_df)
future = m.make_future_dataframe(periods=horizon, include_history=False)
fc = m.predict(future)   # DataFrame with yhat, yhat_lower, yhat_upper

# AutoARIMA — uses statsforecast's long-format DataFrame (needs unique_id column)
from statsforecast import StatsForecast
from statsforecast.models import AutoARIMA
long_df = train_df.assign(unique_id="series")[["unique_id", "ds", "y"]]
sf = StatsForecast(models=[AutoARIMA(season_length=7)], freq="D")
sf.fit(long_df)
fc = sf.predict(h=horizon, level=[95])

Same logic with darts — one shape, two models:

from darts import TimeSeries
from darts.models import Prophet, AutoARIMA

ts = TimeSeries.from_dataframe(train_df, "ds", "y", freq="D")

# Both models share the same fit/predict API
prophet_fc = Prophet(country_holidays="SE").fit(ts).predict(horizon)
arima_fc   = AutoARIMA(season_length=7).fit(ts).predict(horizon)

Naming note — AutoARIMA, not "SARIMA". Darts (and statsforecast) name the class AutoARIMA because its job is to auto-select the (p, d, q, P, D, Q) orders by minimizing AIC. Set season_length=7 and you get SARIMA behavior (weekly seasonal ARIMA); omit it and you get plain ARIMA (no seasonality). The HTML doc says "SARIMA" because that's what the model functionally is in our configuration — but the class name you'd import is AutoARIMA.

Benchmark — raw vs darts, identical setup:

                       RAW           DARTS         DIFF
Prophet            0.08s         0.07s         noise
AutoARIMA          0.51s         0.50s         noise

Prophet MAPE     12.33%        12.33%        0.000%
AutoARIMA MAPE   13.50%        13.50%        0.000%

Identical forecasts, identical speed (both wrap the same engines, no actual difference at runtime). So for the current two-model ensemble we kept the direct calls — they're more transparent for debugging and avoid the extra quantile_timeseries(0.025) step that darts requires for confidence intervals.

When the switch pays off:

  • Adding more model families — if we ensembled NBEATS, RNN, LightGBM, etc. alongside Prophet and AutoARIMA, the uniform TimeSeries API saves real code. Each new model drops in with the same fit() / predict() shape.
  • Built-in backtesting / cross-validation — darts has historical_forecasts() and backtest() out of the box. With raw libraries we'd write those helpers ourselves.

A drop-in darts version of _fit_prophet and _fit_autoarima sits as a commented reference at the bottom of backend/agents/forecasting_agent.py. Swap the two functions with the commented versions and nothing else in the pipeline changes.

VISUALIZATION AGENT

Plotly visualization from code agent rows

Sequential agent — runs after the code agent. Takes the rows and the user's original question, picks the right chart type and columns, and produces a self-contained Plotly HTML figure that the frontend embeds in an iframe.

Input

Three arguments from the supervisor (passed positionally): 
columns:       list[str]      # from the code agent's output
rows:          list[dict]     # from the code agent's output
user_question: str            # the original NL query

Output

{
  "chart_type":  str | None,
  "title":       str | None,
  "caption":     str | None,
  "figure_html": str | None,
  "figure_json": str | None,
  "error":       str | None,
}

12 supported chart types

Comparison
bar, horizontal_bar — top-N rankings, grouped comparisons.
Time series
line, area — auto range slider when is_time_series=True.
Distribution
histogram, box, scatter — spread and outlier analysis.
Composition
pie, treemap, funnel — parts-of-a-whole.
Multi-dimensional
scatter_3d, density_heatmap — 3 numeric KPIs in one view, or 2D density / correlation patterns.

Internal flow — step by step

1
Empty-rows early return
If the upstream SQL returned no rows, the chart agent skips the LLM call entirely and returns a clear error dict. No point wasting tokens on data that doesn't exist.
if not rows:
    return {"chart_type": None, ..., "error": "No data to plot"}
2
get_chart_spec() — LLM picks the chart type and columns
The agent sends the column list, 3 sample rows, and the user's original question to the LLM. Structured output returns a ChartSpec: chart_type (one of 12), the x / y / z column names, optional color or path columns, an is_time_series flag, plus a title and one-line caption.
response = client.chat.completions.parse(
    model="gpt-5-mini",
    messages=[{"role": "system", ...}, {"role": "user", ...}],
    response_format=ChartSpec,
)
3
Validate the spec + retry once on failure
Two checks: every column the LLM picked must actually exist in columns (no hallucinations), and required-fields must match the chart type (e.g., treemap needs path_columns and values_column). If validation fails, the LLM is called again with the error message as feedback — one retry, then give up gracefully.
err = validate_spec(spec, columns)
if err and MAX_RETRIES > 0:
    spec = get_chart_spec(columns, rows, user_question, error_context=err)
    err = validate_spec(spec, columns)
Required fields per chart type — what the validator enforces:
┌────────────────────────────────────────────────────────────┬──────────────────────────────────┐
│ Chart type                                                  │ Required fields                  │
├────────────────────────────────────────────────────────────┼──────────────────────────────────┤
│ bar / horizontal_bar / line / area / scatter / box / funnel │ x_column, y_column               │
│ pie                                                         │ x_column (names), y_column (vals)│
│ histogram                                                   │ x_column only                    │
│ treemap                                                     │ path_columns + values_column     │
│ scatter_3d                                                  │ x_column, y_column, z_column     │
│ density_heatmap                                             │ x_column, y_column (z optional)  │
└────────────────────────────────────────────────────────────┴──────────────────────────────────┘
If the LLM picks treemap but forgets path_columns, validation catches it and retries with the specific error message. Same retry pattern as the SQL agent — one chance to self-correct, never a loop.
4
Render the Plotly figure with the right one-liner
A switch on chart_type picks the right Plotly Express call: px.bar, px.line, px.treemap, etc. Time-series charts get an interactive range slider when is_time_series=True. Output is a self-contained HTML string (Plotly.js loaded from CDN) for the frontend iframe.
if spec.chart_type == "bar":
    fig = px.bar(data_frame=df, x=spec.x_column, y=spec.y_column, title=spec.title)
elif spec.chart_type == "treemap":
    fig = px.treemap(data_frame=df, path=spec.path_columns, values=spec.values_column, ...)
# ... scatter_3d, density_heatmap, line, area, scatter, box, histogram, pie, funnel
if spec.is_time_series:
    fig.update_xaxes(rangeslider_visible=True)
Per-chart-type rendering tweaks — small but important. The render switch isn't just "pick the right function"; each chart type gets one or two Plotly tweaks that make the output read correctly:
# bar — y-axis anchored to zero (avoids misleading "tall but tiny range" bars)
fig = px.bar(...); fig.update_yaxes(rangemode="tozero")

# horizontal_bar — biggest bar at top for natural ranking reads
fig = px.bar(orientation="h", ...); fig.update_yaxes(categoryorder="total ascending")

# treemap — adds an "All" root tile so the hierarchy renders cleanly
fig = px.treemap(path=[px.Constant("All")] + spec.path_columns, ...)

# density_heatmap — averages z by (x,y) bins instead of just counting when z provided
fig = px.density_heatmap(..., z=spec.z_column, histfunc="avg")
5
Pack the result dict and return to the supervisor
The agent returns chart_type, title, caption, the self-contained figure_html and figure_json, plus any error. The supervisor stores it in state["chart_results"]; main.py later pulls figure_html into the ChatResponse so the frontend can embed it.
SYNTHESIZER AGENT

Composing the final briefing

The last agent in the pipeline. Reads every specialist's output and composes a single grounded, citation-aware markdown narrative — this is the text the frontend renders for the user.

Input

The full state dict produced by the supervisor (every specialist's result, or None when that specialist wasn't in the plan): 
state = {
  "user_query":       str,
  "plan":             dict,
  "sql_results":      dict | None,
  "web_results":      dict | None,
  "forecast_results": dict | None,
  "chart_results":    dict | None,
}

Output

{
  "final_report": str | None,   # markdown
  "error":        str | None,
  "skipped":      bool,
}

Internal flow — step by step

1
Smart-skip check — trivial queries don't need synthesis
Some queries don't need narrative wrapping — e.g., a single-row SQL count where the section dump already reads cleanly. The synthesizer's should_skip_synthesis() checks this. Forecast always needs narrative, so it never skips when a forecast is present.
if should_skip_synthesis(state):
    return {"final_report": None, "error": None, "skipped": True}
Skip-conditions table — the four scenarios should_skip_synthesis() evaluates:
┌─────────────────────────────────────────────────┬───────┬────────────────────────────────────┐
│ Scenario                                         │ Skip? │ Reason                             │
├─────────────────────────────────────────────────┼───────┼────────────────────────────────────┤
│ Forecast result present                          │ Never │ Forecasts always need narrative    │
│ Multi-result query (SQL + Web, SQL + Chart, …)   │ Never │ Composing across sources IS the    │
│                                                  │       │   synthesizer's whole job          │
│ Single-row SQL count, nothing else               │  YES  │ "1,371,980 active members" reads   │
│                                                  │       │   fine without wrapping prose      │
│ Anything else                                    │ Never │ Default to synthesizing            │
└─────────────────────────────────────────────────┴───────┴────────────────────────────────────┘
When the synthesizer skips, main.py falls back to a section-dump rendering — each specialist's output gets its own markdown header. So skipping doesn't mean "no answer" — it means "the raw structure is clearer than a paragraph would be."
2
Build the user message — format each specialist's output
Every specialist's full output would blow the LLM's token budget (figure HTML alone is 50KB+). The synthesizer slims each one: SQL gets first 5 rows + row count, web gets the answer + first 6 citations, forecast gets the series label + MAPEs + 3 sample rows from the values table, chart gets just title + caption.
user_message = (
    f"USER QUESTION: {state["user_query"]}\n\n"
    f"SQL DATA:\n{_format_sql_block(state["sql_results"])}\n\n"
    f"WEB RESEARCH:\n{_format_web_block(state["web_results"])}\n\n"
    f"FORECAST:\n{_format_forecast_block(state["forecast_results"])}\n\n"
    f"CHART:\n{_format_chart_block(state["chart_results"])}\n"
)
What each specialist's slim block actually contains — the per-formatter slicing keeps tokens controlled:
┌──────────────┬─────────────────────────────────────────────┬──────────────────────────────────────┐
│ Specialist   │ Full output                                  │ What goes into the LLM block         │
├──────────────┼─────────────────────────────────────────────┼──────────────────────────────────────┤
│ SQL          │ sql, explanation, columns, rows (up to 1000),│ Explanation + row_count + sample of  │
│              │   row_count, error                            │   first 5 rows                       │
│ Web          │ answer (markdown), citations (list),         │ Answer text + first 6 citations      │
│              │   search_queries_run, model, error            │                                      │
│ Forecast     │ historical (long array), prophet_forecast,   │ Series label + horizon + MAPEs +     │
│              │   sarima_forecast, ensemble_forecast,         │   first/mid/end 3-row sample of      │
│              │   metrics, chart_html (50KB+), values_table   │   values_table                       │
│ Chart        │ chart_type, title, caption, figure_html,     │ Title + caption only                 │
│              │   figure_json                                 │   (HTML is sent to frontend          │
│              │                                               │    separately, not via LLM)          │
└──────────────┴─────────────────────────────────────────────┴──────────────────────────────────────┘
Why slim? If the full figure_html (50KB) or full historical array (thousands of rows) went into the LLM message, the prompt token cost would explode and the model's effective attention would be diluted on irrelevant detail. The slim version preserves the decision-relevant signal: what the question was, what the data says in summary, what each specialist concluded. Each formatter also has a fallback: if a specialist returned None, the block becomes "(none)"; if it returned an error, the block becomes "(error: <message>)" — consistent shape regardless of plan composition.
3
Single LLM call — returns free-form markdown narrative
Unlike most other agents, the synthesizer does NOT use structured outputs — it returns prose. The system prompt enforces style rules (lead with the answer, embed citations inline, translate stat jargon, never write a "### Chart" header, etc.). Errors are caught and returned as error in the result dict.
response = client.chat.completions.create(
    model="gpt-5-mini",
    messages=[{"role": "system", ...}, {"role": "user", ...}],
)
final_report = response.choices[0].message.content
4
Pack result and return to the supervisor
The result dict has three fields: final_report (markdown), error, skipped (bool). Supervisor stores them in state["final_report"], state["synthesizer_skipped"], state["synthesizer_error"]. main.py then sends the markdown to the frontend as the text field of ChatResponse — or, if synthesis was skipped/errored, it falls back to a section-dump format built from the raw specialist outputs.

Adding session-level memory (optional)

The pipeline is stateless by default. main.py reads only the latest user message and ignores the rest of the conversation, so follow-up questions like “now break that down by category” cannot resolve pronouns or references. To enable session memory (memory that lasts as long as the browser tab is open), pass the full message history into the supervisor as context. One small change in main.py is enough — no agent needs to be modified.

The one-line switch inside chat()

Today the handler reads only the latest message. Replace that one line with a block that concatenates the full history into a single string, and the supervisor sees the whole conversation when it builds the plan.

Current (stateless) — this is the line to replace:

user_query = user_msgs[-1].content

With session memory — replace the line above with this block:

history = "\n".join(
    f"{m.role.upper()}: {m.content}" for m in req.messages[:-1]
)
current = req.messages[-1].content
user_query = (
    f"Conversation so far:\n{history}\n\nCurrent question: {current}"
    if history else current
)

After this change, the supervisor sees the prior turns when it builds the plan, so it can resolve references against the conversation. Memory lives only inside the request payload — it lasts as long as the tab is open. Closing the tab or refreshing the page clears it. The replacement block also sits at the bottom of main.py as a commented reference, so enabling memory is uncomment-and-swap.

Long-term memory across sessions (LangGraph checkpointer)

Session memory dies when the tab closes. For conversations that survive across sessions, browser refreshes, and even server restarts, use LangGraph's built-in checkpointer. The LangGraph supervisor variant supervisor_L.py is already wired for this — adding the SQLite checkpointer is a small change. Every plan, every specialist result, and every final report is persisted per thread_id and auto-loaded on the next request for that thread.

The change in supervisor_L.py

from langgraph.checkpoint.sqlite import SqliteSaver

CHECKPOINT_DB = BACKEND_DIR / "data" / "db" / "checkpoints.db"

def build_graph():
    g = StateGraph(AgentState)
    # ... existing node and edge setup ...
    checkpointer = SqliteSaver.from_conn_string(str(CHECKPOINT_DB))
    return g.compile(checkpointer=checkpointer)

def run(user_query: str, thread_id: str) -> dict:
    app = build_graph()
    config = {"configurable": {"thread_id": thread_id}}
    return app.invoke(initial_state, config=config)

The change in main.py

class ChatRequest(BaseModel):
    messages: list[Message]
    thread_id: str   # NEW — sent by the frontend (localStorage UUID)

@app.post("/api/chat")
async def chat(req: ChatRequest) -> ChatResponse:
    user_query = req.messages[-1].content
    state = await asyncio.to_thread(run_pipeline, user_query, req.thread_id)

The frontend generates a UUID once on first load, stores it in localStorage, and sends it with every request. From that point on, the supervisor sees all prior state for the thread — prior plans, prior SQL rows, prior final reports — and the planner can reference them when building the next plan. Different users (different thread_ids) have isolated memories.

Deployment story — no extra database server

The checkpointer DB is just a SQLite file on the server's local disk — the same storage model as hm.duckdb. SQLite is embedded: no separate database server, no managed service, no extra infra. It ships as a Python package (pip install langgraph-checkpoint-sqlite) that writes to whatever file path you give it. On deployment, the checkpoints.db file lives alongside hm.duckdb in backend/data/db/ — same disk, same backup story, zero new moving parts.

If you later scale to multiple server instances behind a load balancer, swap SqliteSaver for PostgresSaver and point it at any Postgres connection string (a managed service like Supabase / Neon / RDS, or your own Postgres container). The supervisor code does not change — only the checkpointer instantiation line. Same pattern as swapping DuckDB for BigQuery in production.
